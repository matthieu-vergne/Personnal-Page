Agile Refactoring
or How to Change without Breaking

Context

Refactoring is the process of changing the code without changing the behaviour. It is used to improve the code quality on aspects which do not affect the features of the application. Improving the code readability or giving it a simpler structure are some aspects that are generally considered while refactoring code. By remaining isofunctional, the program should still compile and functional tests should still pass once the refactoring is finished.

Problem: Refactoring is necessarily costly, but not necessarily valuable

In articles like the one you are reading, we often use simple examples which focus on the aspects we want to illustrate. Consequently, it is often easy to pass from a "bad" code to a "better" one. However, in real cases, it often requires (a lot) more time and effort. The code is longer, dirtier, we need more time to think about what to do, discuss with the team, etc. Personally, I spend some time implementing something that works, but more time refactoring it. Often because it is only once you have something working that you can really see all the components you need and refactor them in a clear way.

Moreover, refactoring is isofunctional, which means that it adds no feature at all. You spend some time adding no value to the software, but expect to make it easier to increase this value after refactoring to compensate. In other words, it must be considered as an investment: you necessarily lose on the short term while expecting more gains on a longer term. It is often hard to motivate refactoring in a business setting with such a speculative aspect.

Refactoring Methods

Agile practices organize the work in order to quickly deliver the produced software. By improving organizational and coding practices, people should be able to focus on what is the most important at the current time. This focus allow to spend the effort on the most important changes, thus increasing the value of the software as much as possible given the limited resources. By repeating it in small iterations, it ensures that we regularly improve the software.

As such, agility goes way beyond the code itself, but we will focus on how to apply this logics to code refactoring. In other words, we aim at adapting the code while remaining isofunctional at each and every step of the refactoring. It allows you to refactor the code progressively, spending only the time you can spend on this task. You can freely stop refactoring, do something with higher priority, and come back to it when you can. By splitting the refactoring task into arbitrarily small pieces, you can more easily spend time on it despite the other things to do.

In this article, we focus on manual procedures to refactor your code. IDEs often support some, if not all, of these procedures. If you use one, I strongly recommend you to check its documentation to see how to do them quickly. IDEs are automated, which means they tend to systematically and correctly execute the changes, thus avoiding errors prone to occur with manual procedures. You can also gain a lot in coding speed, which saves your time for more interesting things to do.

Refactoring variables

Extract a variable

Starting from an instruction which consumes a value:


	System.out.println("foo");

Create a variable with the value:


	String myString = "foo";
	System.out.println("foo");

Use the variable instead of the value:


	String myString = "foo";
	System.out.println(myString);// Replace it everywhere required

If the value is a method call, check whether it has side effects. If it has, extracting several calls into a single variable will change the behaviour of the program.

Inline a variable

Starting from an instruction which consumes a variable:


	String myString = "foo";
	System.out.println(myString);

Inline the variable by applying directly its value:


	String myString = "foo";
	System.out.println("foo");// Replace it everywhere required

You can move the variable down in the code or remove it if nobody uses it anymore:


	System.out.println("foo");

If the value is a method call, check whether it has side effects. If it has, inlining a single variable into several calls will change the behaviour of the program.

Split a variable declaration

Starting from a single instruction which declares and instantiate your object:


	Object myObject = new Object();

Separate the instantiation of the object with a second instruction:


	Object myObject;
	myObject = new Object();

Join a variable declaration

Starting from an instruction which declare your variable and another which instantiates it:


	Object myObject;
	myObject = new Object();

Join them into a single instruction:


	Object myObject = new Object();

Increase a variable scope

Starting from a variable declaration:


	if (checkIsOk) {
		Object myObject;
		// myObject usable only here
	}

Increase its scope by moving it higher in the blocks:


	Object myObject;
	if (checkIsOk) {
		// myObject usable here
	}
	// myObject usable also here

You may already have a variable with the same name in the higher scope. Rename one of the variables before to change the scope to avoid conflicts.

Decrease a variable scope

Starting from a variable declared out of a block:


	Object myObject;
	if (checkIsOk) {
		// myObject usable here
	}
	// myObject usable also here

Decrease its scope to the block only:


	if (checkIsOk) {
		Object myObject;
		// myObject usable only here
	}

The variable might be used out of the block you target. You need to remove these uses or replace them by other ones. For replacements, you can for example create a separate variable or inline the variable there.

Rename a variable

Starting from an existing variable:


	String myAwfulName = "foo";
	System.out.println(myAwfulName);

Create a new variable with the name you want:


	String myAwfulName = "foo";
	String myAwsomeName;
	System.out.println(myAwfulName);

Assign it the current variable:


	String myAwfulName = "foo";
	String myAwsomeName = myAwfulName;
	System.out.println(myAwfulName);

Replace the old variable by the new variable where it is used:


	String myAwfulName = "foo";
	String myAwsomeName = myAwfulName;
	System.out.println(myAwsomeName);// Replace it everywhere required

Inline the old variable once it is only used by the new variable:


	String myAwsomeName = "foo";
	System.out.println(myAwsomeName);
