Agile Refactoring
or How to Change without Breaking

Context

Refactoring is the process of changing the code without changing the behaviour. It is used to improve the code quality on aspects which do not affect the features of the application. Improving the code readability or giving it a simpler structure are some aspects that are generally considered while refactoring code. By remaining isofunctional, the program should still compile and functional tests should still pass once the refactoring is finished.

Problem: Refactoring is necessarily costly, but not necessarily valuable

In articles like the one you are reading, we often use simple examples which focus on the aspects we want to illustrate. Consequently, it is often easy to pass from a "bad" code to a "better" one. However, in real cases, it often requires (a lot) more time and effort. The code is longer, dirtier, we need more time to think about what to do, discuss with the team, etc. Personally, I spend some time implementing something that works, but more time refactoring it. Often because it is only once you have something working that you can really see all the components you need and refactor them in a clear way.

Moreover, refactoring is isofunctional, which means that it adds no feature at all. You spend some time adding no value to the software, but expect to make it easier to increase this value after refactoring to compensate. In other words, it must be considered as an investment: you necessarily lose on the short term while expecting more gains on a longer term. It is often hard to motivate refactoring in a business setting with such a speculative aspect.

Method

Agile practices organize the work in order to quickly deliver the produced software. By improving organizational and coding practices, people should be able to focus on what is the most important at the current time. This focus allow to spend the effort on the most important changes, thus increasing the value of the software as much as possible given the limited resources. By repeating it in small iterations, it ensures that we regularly improve the software.

As such, agility goes way beyond the code itself, but we will focus on how to apply this logics to code refactoring. In other words, we aim at adapting the code while remaining isofunctional at each and every step of the refactoring. It allows you to refactor the code progressively, spending only the time you can spend on this task. You can freely stop refactoring, do something with higher priority, and come back to it when you can. By splitting the refactoring task into arbitrarily small pieces, you can more easily spend time on it despite the other things to do.
